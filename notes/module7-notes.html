Securing the Network: An Integrated View

Network security is not a single protocol but a layered collection of technologies that protect communication across different layers of the TCP/IP stack. In this module we examine security at the application layer (email, web), the transport layer, and the network layer – all built on the cryptographic primitives studied in earlier modules.

Authentication Services: Kerberos and X.509

Kerberos

Kerberos is a network authentication protocol that uses symmetric key cryptography and a trusted third party – the Key Distribution Center (KDC). It is designed to provide strong authentication in open, untrusted networks.

  • Authentication Server (AS) – verifies the user’s identity (e.g., via password) and issues a Ticket‑Granting Ticket (TGT).
  • Ticket‑Granting Server (TGS) – exchanges the TGT for a service ticket that the client presents to a specific service.
  • The service ticket contains a session key encrypted for the service, enabling mutual authentication and optional message encryption.

Kerberos version 5 is widely used in enterprise environments (Active Directory) and supports PKINIT for initial authentication using public‑key certificates instead of a password.

X.509 Authentication Service

X.509 defines the format for public‑key certificates and a framework for certificate validation. Each certificate contains:

  • Subject’s name and public key.
  • Issuer (Certificate Authority) name.
  • Validity period.
  • Digital signature of the CA over the certificate body.

Certificate validation involves building a chain of trust up to a trusted root CA, checking signatures, expiration, and revocation status via Certificate Revocation Lists (CRLs) or the Online Certificate Status Protocol (OCSP).

Electronic Mail Security

Pretty Good Privacy (PGP)

PGP provides end‑to‑end encryption, authentication, and compression for email. It uses a hybrid cryptosystem:

  • A random symmetric session key encrypts the message body (using AES, IDEA, or CAST5).
  • The session key is encrypted with the recipient’s RSA or ElGamal public key.
  • Digital signatures use the sender’s private key over a hash of the message.

PGP follows a web of trust model – users can sign each other’s keys, and trust is built through social verification rather than a central CA. OpenPGP (RFC 4880) is the open standard, implemented by GnuPG.

S/MIME (Secure/Multipurpose Internet Mail Extensions)

S/MIME is an IETF standard that adds security to MIME messages (email with attachments). Unlike PGP, S/MIME relies on a hierarchical PKI (X.509 certificates). It provides:

  • Enveloped data – symmetric encryption (AES) with a key wrapped in RSA.
  • Signed data – digital signature (RSA, DSA, ECDSA) over a message hash.
  • Clear‑signed data – signature only, readable by non‑S/MIME clients.

S/MIME is the dominant standard in enterprise environments where centralised certificate management is feasible.

IP Security (IPsec)

IPsec provides security at the network layer, protecting all traffic between two endpoints or networks. It is transparent to applications and transport protocols.

IPsec Architecture

  • Security Association (SA) – a unidirectional logical connection specifying the algorithms and keys for traffic. Identified by a triplet (SPI, destination IP, protocol).
  • Security Association Database (SAD) – stores active SAs.
  • Security Policy Database (SPD) – defines which traffic is to be protected and how (bypass, discard, or protect).

IPsec Protocols

  • Authentication Header (AH) – provides data integrity and origin authentication but no encryption. Protects the entire packet including immutable IP header fields.
  • Encapsulating Security Payload (ESP) – provides confidentiality, and optionally authentication. Encrypts the payload and can also authenticate portions of the packet.

Transport mode – encrypts/authenticates only the IP payload (for end‑to‑end security between hosts). Tunnel mode – encapsulates the entire original IP packet and adds a new IP header (used for VPN gateways).

Key Management in IPsec

The Internet Key Exchange (IKEv2) protocol negotiates SAs and establishes shared keys. IKEv2 uses a Diffie‑Hellman exchange within a secure channel (created via initial IKE_SA_INIT and IKE_AUTH exchanges). It supports mutual authentication via pre‑shared keys, digital signatures, or EAP.

Web Security: SSL/TLS

Transport Layer Security (TLS), the successor to SSL, is the most widely used security protocol. It sits above TCP and provides a secure channel for higher‑layer protocols (HTTPS, SMTPS, LDAPS).

TLS Protocol Layers

  • Handshake Protocol – negotiates session parameters: protocol version, cipher suite, and authentication. Uses public‑key cryptography to establish a pre‑master secret from which symmetric session keys are derived.
  • Record Protocol – fragments, compresses (optional), applies a MAC, and encrypts data using the agreed‑upon symmetric algorithms (e.g., AES‑GCM, ChaCha20‑Poly1305).

TLS Handshake Overview (simplified)

  1. Client → Server: ClientHello (supported versions, cipher suites, random nonce).
  2. Server → Client: ServerHello (chosen parameters), Certificate, ServerKeyExchange (if needed), ServerHelloDone.
  3. Client → Server: ClientKeyExchange (encrypted pre‑master secret), ChangeCipherSpec, Finished (first encrypted message).
  4. Server → Client: ChangeCipherSpec, Finished.

After the handshake, all traffic is encrypted and authenticated with symmetric keys derived from the pre‑master secret.

Security Requirements for Web Transactions

  • Confidentiality – encrypted channel.
  • Authentication – server (always), optionally client.
  • Integrity – MAC prevents tampering.
  • Forward secrecy – even if the server’s long‑term key is later compromised, past sessions remain secure (achieved by ephemeral Diffie‑Hellman key exchange).
💡 TLS 1.3 significantly simplifies the handshake (one round‑trip), mandates forward secrecy, removes legacy algorithms (RSA key transport, static DH), and encrypts most of the handshake.

Practical Example (TLS Connection in Java)

import javax.net.ssl.*;
import java.io.*;

public class TLSDemo {
    public static void main(String[] args) throws Exception {
        SSLSocketFactory factory = (SSLSocketFactory) SSLSocketFactory.getDefault();
        SSLSocket socket = (SSLSocket) factory.createSocket("example.com", 443);
        socket.startHandshake();

        PrintWriter out = new PrintWriter(socket.getOutputStream(), true);
        BufferedReader in = new BufferedReader(new InputStreamReader(socket.getInputStream()));

        out.println("GET / HTTP/1.1\r\nHost: example.com\r\n\r\n");
        String line;
        while ((line = in.readLine()) != null) {
            System.out.println(line);
        }
        socket.close();
    }
}

Summary

This module covered the major security protocols at the application, transport, and network layers: Kerberos and X.509 for authentication, PGP and S/MIME for email, IPsec for network‑layer VPNs, and TLS for web security. Together, they form the backbone of a secure Internet. The next module will address the unique challenges of wireless environments.
