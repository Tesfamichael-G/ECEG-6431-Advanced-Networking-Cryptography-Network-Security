The Need for Public Key Cryptography

Symmetric encryption requires both parties to share a secret key beforehand – a major challenge in large networks. Public key cryptography (asymmetric cryptography) solves this by using a key pair: a public key (known to everyone) and a private key (kept secret). The two keys are mathematically related, but it is computationally infeasible to derive the private key from the public key.

This enables confidentiality (encrypt with public key, decrypt with private key), authentication and non‑repudiation via digital signatures (sign with private key, verify with public key), and key exchange (agree on a shared secret without prior secrets).

RSA (Rivest–Shamir–Adleman) Cryptography

RSA is the most widely used public‑key algorithm, introduced in 1977. Its security is based on the difficulty of factoring large integers.

Key Generation

  1. Choose two large, distinct primes p and q (typically 1024–2048 bits each).
  2. Compute the modulus: n = p · q.
  3. Compute Euler’s totient: φ(n) = (p–1)(q–1).
  4. Choose a public exponent e such that 1 < e < φ(n) and gcd(e, φ(n)) = 1. Common choices are 65537 (0x10001) for efficiency.
  5. Compute the private exponent d as the modular inverse of e modulo φ(n): d ≡ e–1 (mod φ(n)). This is done using the Extended Euclidean algorithm.
  6. Public key: (e, n). Private key: (d, n) (or (d, p, q) for CRT speedup).

Encryption and Decryption

To encrypt a message m (represented as an integer m < n): 

c ≡ me (mod n)

To decrypt: 

m ≡ cd (mod n)

The correctness follows from Euler’s theorem: since d ≡ e–1 (mod φ(n)), we have e·d = 1 + k·φ(n) for some integer k, so cd ≡ (me)d = m1 + k·φ(n) ≡ m·(mφ(n))k ≡ m (mod n).

⚠️ Practical RSA: Messages are never encrypted directly. Instead, a random symmetric session key is encrypted with RSA, and the actual data is encrypted under that session key (hybrid encryption). Also, padding schemes like OAEP are mandatory to prevent attacks.

Elliptic Curve Cryptography (ECC)

ECC offers the same security as RSA with much smaller key sizes and faster computations. It is based on the algebraic structure of elliptic curves over finite fields.

An elliptic curve over a prime field GF(p) is defined by the equation y2 = x3 + a·x + b (mod p), with the curve parameters (p, a, b, a generator point G, and order n). The set of points, together with a point at infinity O, forms an abelian group under a geometrically defined addition operation.

The security of ECC relies on the Elliptic Curve Discrete Logarithm Problem (ECDLP): given points P and Q where Q = k·P, it is computationally infeasible to find the scalar k.

ECC Key Exchange and Encryption

  • ECDH – Elliptic Curve Diffie‑Hellman key exchange.
  • ECIES – Elliptic Curve Integrated Encryption Scheme (provides confidentiality).
  • ECDSA – Elliptic Curve Digital Signature Algorithm.

For the same security level, an ECC key of 256 bits is comparable to an RSA key of 3072 bits, making ECC ideal for mobile devices and IoT.

Key Management and the Diffie‑Hellman Key Exchange

Key management encompasses the generation, storage, distribution, rotation, and destruction of cryptographic keys. In public‑key systems, the secure distribution of public keys is achieved via Public Key Infrastructure (PKI) and digital certificates.

Diffie‑Hellman Key Exchange Protocol

Invented by Whitfield Diffie and Martin Hellman in 1976, this protocol enables two parties to agree on a shared secret over an insecure channel without any prior secrets.

  1. Alice and Bob agree on public parameters: a large prime p and a generator g of the multiplicative group modulo p.
  2. Alice chooses a random private key a (1 < a < p‑1) and computes A ≡ ga (mod p). She sends A to Bob.
  3. Bob chooses a random private key b and computes B ≡ gb (mod p). He sends B to Alice.
  4. Alice computes the shared secret: s ≡ Ba (mod p).
  5. Bob computes: s ≡ Ab (mod p).

Both end up with the same value because Ba ≡ (gb)a ≡ gab ≡ (ga)b ≡ Ab (mod p). The beauty: an eavesdropper who sees p, g, A, B cannot compute gab without solving the discrete logarithm problem.

💡 Man‑in‑the‑Middle Warning: Basic DH is secure against passive eavesdropping but vulnerable to active attacks. To prevent this, the exchanged values A and B must be authenticated (e.g., using digital signatures or a PKI).

Key Management Challenges

  • Key generation – must use cryptographically secure random number generators (CSPRNGs).
  • Key storage – private keys must be protected, often using hardware security modules (HSMs).
  • Key distribution – public keys are distributed via certificates and PKI; symmetric keys via key exchange protocols.
  • Key revocation – certificates may need to be revoked if keys are compromised; this is managed by Certificate Revocation Lists (CRLs) or the Online Certificate Status Protocol (OCSP).
  • Key rotation – regularly updating keys limits the damage of a potential breach.

Comparing RSA and ECC

Aspect RSA ECC
Underlying hard problem Integer factorisation Elliptic curve discrete log
Key size (128‑bit security) 3072 bits 256 bits
Performance Slower key generation; faster encryption (small public exponent) Faster overall, especially key gen and signing
Use cases Legacy systems, TLS, PGP Mobile, IoT, blockchain, modern TLS

Mathematical Example (RSA)

Let p=3, q=11. Then n=33, φ(n)=20. Choose e=7. Compute d = 7–1 mod 20 = 3. Encrypt m=2: c = 27 mod 33 = 128 mod 33 = 29. Decrypt: 293 mod 33 = 24389 mod 33 = 2 (since 24389 = 739*33 + 2).
