From Handwritten to Digital Signatures

A digital signature is a cryptographic mechanism that provides the digital equivalent of a handwritten signature – but far stronger. It assures the recipient of the authenticity of the sender, the integrity of the message, and provides non‑repudiation (the signer cannot later deny having signed). Digital signatures are a cornerstone of secure electronic commerce, software distribution, and digital identity.

How Digital Signatures Work

Digital signatures use asymmetric cryptography. The signer uses their private key to create the signature; anyone with access to the signer's public key can verify it. The general process involves hashing the message first, then encrypting the hash with the private key. 

// Signing
hash = H(message)
signature = SignprivateKey(hash)

// Verification
hash1 = H(message)
hash2 = VerifypublicKey(signature)
if (hash1 == hash2) → valid

RSA Digital Signature Scheme

RSA can be used directly for signing by reversing the roles of encryption and decryption: the signature s is computed as s ≡ md mod n, where d is the private key. However, raw RSA signing is insecure. The standardised scheme is RSA‑PSS (Probabilistic Signature Scheme), which adds random padding and is provably secure in the random oracle model.

  • RSASSA‑PKCS1‑v1_5 – older scheme, still widely used, but not as robust as PSS.
  • RSASSA‑PSS – recommended for new applications; includes a random salt and mask generation.

Digital Signature Algorithm (DSA)

DSA is a US federal standard (FIPS 186) based on the discrete logarithm problem. It generates two values, r and s, that together form the signature. DSA requires a per‑message random nonce (k), and re‑using the same nonce with different messages immediately reveals the private key! 

// DSA parameters: p, q, g (public); x (private); y = gx mod p (public)

Sign:
  Choose random k (1 < k < q)
  r = (gk mod p) mod q
  s = k-1(H(m) + x·r) mod q
  Signature = (r, s)

Verify:
  w = s-1 mod q
  u1 = H(m)·w mod q
  u2 = r·w mod q
  v = (gu1·yu2 mod p) mod q
  Valid if v == r

Elliptic Curve Digital Signature Algorithm (ECDSA)

ECDSA is the elliptic curve analogue of DSA, offering the same functionality with smaller keys and faster operations. It is the basis of signature verification in Bitcoin, Ethereum, and modern TLS (with curves like secp256r1, Curve25519). The core idea is to replace modular exponentiation with point multiplication on an elliptic curve.

Authentication Protocols

An authentication protocol enables one entity (the claimant) to prove its identity to another (the verifier). These protocols must resist replay attacks, impersonation, and man‑in‑the‑middle attacks.

Challenge‑Response Protocols

The verifier sends a random nonce (number used once) to the claimant. The claimant processes the nonce (e.g., encrypts it with a shared key or signs it with a private key) and returns the result. Because the nonce is fresh and unpredictable, the response cannot be replayed. 

Verifier → Claimant: N (random nonce)
Claimant → Verifier: EK(N)   // or Signprivate(N)
Verifier checks if decrypted N matches original

Kerberos

Kerberos is a network authentication protocol designed at MIT. It uses a trusted third party, the Key Distribution Center (KDC), to issue tickets. A ticket contains a session key encrypted for a specific service, allowing mutual authentication and session encryption.

  • Authentication Server (AS) – authenticates the user and issues a Ticket‑Granting Ticket (TGT).
  • Ticket‑Granting Server (TGS) – exchanges the TGT for a service ticket.
  • Uses symmetric cryptography (DES, later RC4‑HMAC, now AES). Modern Kerberos also supports PKINIT for public‑key initial authentication.

X.509 Authentication Service

X.509 defines a framework for authentication using public‑key certificates and a hierarchical Certificate Authority (CA) system. Each certificate binds a public key to a distinguished name. The certificate is signed by a CA, forming a chain of trust up to a self‑signed root certificate. X.509 is the backbone of TLS/SSL, S/MIME, and IPsec.

  • One‑way authentication – only the server (or one party) proves its identity.
  • Two‑way (mutual) authentication – both parties present certificates.
  • Three‑way authentication – adds a random number exchange for additional assurance.

Digital Signature Standards

The Digital Signature Standard (DSS) (FIPS 186) originally specified DSA and later incorporated RSA and ECDSA. Key requirements include:

  • Approved hash functions (SHA‑2 family).
  • Approved elliptic curves (NIST P‑256, P‑384, P‑521; Edwards curves like Ed25519 are also widely adopted).
  • Key sizes: at least 2048 bits for RSA, 256 bits for ECC.

Practical Example (RSA‑PSS in Java)

import java.security.*;

public class RSAPSSExample {
    public static void main(String[] args) throws Exception {
        KeyPairGenerator keyGen = KeyPairGenerator.getInstance("RSA");
        keyGen.initialize(2048);
        KeyPair pair = keyGen.generateKeyPair();

        Signature signer = Signature.getInstance("RSASSA-PSS");
        signer.initSign(pair.getPrivate());
        signer.update("message".getBytes());
        byte[] signature = signer.sign();

        Signature verifier = Signature.getInstance("RSASSA-PSS");
        verifier.initVerify(pair.getPublic());
        verifier.update("message".getBytes());
        boolean valid = verifier.verify(signature);
        System.out.println("Valid: " + valid);
    }
}

Best Practices and Pitfalls

  • Use a secure random nonce for DSA/ECDSA – never reuse, never predict.
  • Always hash the message before signing; sign the hash, not the raw message.
  • Validate certificate chains completely – check expiration, revocation, and correct CA signature.
  • Protect private keys with hardware security modules (HSMs) or trusted execution environments.
  • Avoid deprecated algorithms: MD5, SHA‑1, RSA with PKCS#1 v1.5 (prefer PSS).
