Why Message Authentication Matters

While confidentiality protects data from unauthorised disclosure, it does not guard against tampering or impersonation. Message authentication provides assurance that a message comes from the alleged source (data origin authentication) and has not been altered in transit (integrity). In many security protocols, authentication is even more critical than encryption.

Authentication Requirements and Functions

A secure authentication mechanism must satisfy these requirements:

  • Verification – the recipient can confirm the sender’s identity and message integrity.
  • Non‑forgeability – an attacker cannot construct a valid authentication tag for a new message.
  • Non‑repudiation (optional) – the sender cannot deny having sent the message (requires asymmetric techniques).
  • Timeliness – the message is fresh and not a replay of an earlier transmission.

The three main classes of authentication functions are: message encryption (the ciphertext itself serves as authenticator), message authentication codes (MACs), and hash functions (often combined with keys or digital signatures).

Message Authentication Code (MAC)

A MAC is a short, fixed‑size tag generated from a message and a secret key. It is sent alongside the message. The receiver, who shares the same key, recomputes the MAC and compares it with the received tag. If they match, the message is authentic and unaltered. 

MAC = C(K, M)       // where C is the MAC algorithm, K the secret key, M the message

Unlike a digital signature, a MAC does not provide non‑repudiation because both parties possess the same key. However, MACs are faster and use smaller keys, making them ideal for securing communication between two trusted entities (e.g., in IPsec or TLS record layer).

Cryptographic Hash Functions

A hash function maps an arbitrary‑length input to a fixed‑length output, called the hash value or message digest. Cryptographic hash functions must satisfy three core security properties:

  • Pre‑image resistance (one‑way) – given h = H(x), it is infeasible to find any x’ such that H(x’) = h.
  • Second pre‑image resistance – given x, it is infeasible to find x’ ≠ x such that H(x’) = H(x).
  • Collision resistance – it is infeasible to find any pair x ≠ y such that H(x) = H(y).

Collision resistance implies second pre‑image resistance but is the hardest to achieve. The birthday paradox means a collision can be found in about 2n/2 operations for an n‑bit hash; thus modern hashes use at least 256‑bit outputs.

💡 Hash functions are the workhorses of cryptography: they are used for digital signatures (by hashing the message first), password storage, integrity checks, blockchain consensus, and as building blocks for MACs and key derivation functions.

Security of Hash Functions and MACs

Hash functions face attacks such as brute‑force collision search (birthday attack), differential cryptanalysis, and length‑extension attacks (applicable to Merkle–Damgård constructions like MD5, SHA‑1, SHA‑256). SHA‑3 was designed to resist length‑extension by using a sponge construction.

MAC security relies on both the underlying cryptographic primitive and the construction method. A secure MAC must resist:

  • Existential forgery under chosen‑message attack (EU‑CMA) – even after querying many message‑MAC pairs, an attacker cannot forge a valid MAC for a new message.
  • Key recovery attacks – it should be infeasible to extract the secret key.

Important Hash and MAC Algorithms

MD5 (Message Digest 5)

Produces a 128‑bit hash. Once widely used, it is now cryptographically broken – collisions can be found in seconds on commodity hardware. Never use MD5 for security‑critical applications.

SHA Family (Secure Hash Algorithm)

  • SHA‑1 – 160‑bit output. Collisions found (SHAttered attack, 2017). Deprecated.
  • SHA‑2 – includes SHA‑224, SHA‑256, SHA‑384, SHA‑512. No known practical attacks. SHA‑256 is the current gold standard for many protocols (TLS, Bitcoin).
  • SHA‑3 – Standardised in 2015. Based on the Keccak sponge construction. Complements SHA‑2 rather than replacing it; offers additional resistance against certain structural attacks.

HMAC (Hash‑based Message Authentication Code)

HMAC constructs a secure MAC from an iterative hash function like SHA‑256. The standard is defined in RFC 2104. The construction is: 

HMAC(K, m) = H((K’ ⊕ opad) || H((K’ ⊕ ipad) || m))

where ipad and opad are fixed padding constants, and K’ is the key padded to the block size. HMAC is provably secure under reasonable assumptions about the underlying hash function.

CMAC and GMAC

  • CMAC (Cipher‑based MAC) – based on symmetric block ciphers (e.g., AES). Useful in environments where only a cipher primitive is available.
  • GMAC (Galois Message Authentication Code) – associated with AES‑GCM authenticated encryption, providing both confidentiality and integrity.

Password Hashing

Storing passwords requires special hash functions that are slow and salted to resist brute‑force and rainbow‑table attacks. Notable algorithms: bcrypt, scrypt, and Argon2 (the winner of the Password Hashing Competition).

Practical Example (HMAC‑SHA256)

import javax.crypto.Mac;
import javax.crypto.spec.SecretKeySpec;
import java.util.Base64;

public class HMACDemo {
    public static void main(String[] args) throws Exception {
        String secret = "my-secret-key";
        String message = "important data";

        Mac mac = Mac.getInstance("HmacSHA256");
        SecretKeySpec keySpec = new SecretKeySpec(secret.getBytes(), "HmacSHA256");
        mac.init(keySpec);
        byte[] tag = mac.doFinal(message.getBytes());

        System.out.println(Base64.getEncoder().encodeToString(tag));
    }
}

Common Pitfalls and Best Practices

  • Never roll your own crypto – use standard, well‑vetted libraries.
  • Do not use MD5 or SHA‑1 for security.
  • Always use a salt when hashing passwords or keys; it makes pre‑computed tables infeasible.
  • Use HMAC, not simple hash concatenation – constructions like H(key || message) are vulnerable to length‑extension attacks.
  • Use an appropriate tag size – truncating an HMAC to 128 bits is usually safe; don't use too short a tag.
